# 灰度切换说明

当前仓库已经补齐 Python 服务、测试脚本、回归脚本和部署示例。真正的灰度切换需要在测试环境和网关层执行。

<!-- 下方章节继续展开当前文档的一个核心主题，阅读时建议结合实际代码一起看。 -->
## 1. 前置条件

- Python 服务在开发环境已可稳定启动
- `scripts/smoke_test.py` 已通过
- `scripts/regression_compare.py` 对确定性工具接口比对通过
- `deploy/pre_switch_check.ps1` 可在切流前串行执行烟雾、性能、稳定性、回归检查
- Java 服务保持可启动
- MySQL / Redis 使用与 Java 一致的数据副本或同一测试库

## 2. 推荐灰度顺序

1. 开发环境单独跑通 Python
2. 测试环境接入真实数据库副本
3. 测试环境执行回归与性能探针
4. 小流量灰度到 Python
5. 观察稳定后再切到默认路径
6. Java 保留一段观察期

## 3. 推荐切流方式

### 方式 A：按 Header 切流

请求头：

```http
X-Agent-Version: python
```

适合：

- 研发自测
- 前端白名单验证
- 灰度期间快速切回

### 方式 B：按路径切流

- Java 继续提供 `/agent/chat`
- Python 临时提供 `/py/agent/chat`

适合：

- 不方便改 Header 的调用方
- 需要同时保留两套稳定入口

## 4. 验证清单

灰度前至少确认：

- `/health` 正常
- `/agent/chat` 可返回稳定结果
- `/agent/chat/stream` 首 token 时间可接受
- Redis 命中正常
- 会话记忆能正确读写
- 鉴权角色限制符合预期
- 图表 JSON 前缀仍是 `CHART_JSON:`
- `/ops/metrics` 中 MySQL、Redis、首 token 指标处于可接受范围

## 5. 灰度观察指标

建议重点看：

- Python `/agent/chat` 平均响应时间
- Python `/agent/chat/stream` 首 token 时间
- 5xx 比例
- 模型网关报错比例
- Redis 命中率
- MySQL 查询耗时
- 会话记忆写入成功率

## 7. 灰度完成标准

满足以下条件后再考虑全量：

- 连续观察周期内无明显 5xx 异常
- 回归脚本输出稳定
- 业务方确认核心问句结果可接受
- 与 Java 版相比无明显功能缺失
