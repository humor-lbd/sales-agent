# 回滚说明

即使 Python 版已经跑通，也必须把回滚作为正式切换的一部分准备。

<!-- 下方章节继续展开当前文档的一个核心主题，阅读时建议结合实际代码一起看。 -->
## 1. 触发条件

出现下列情况之一时，建议立即回滚：

- `/agent/chat` 或 `/agent/chat/stream` 持续 5xx
- 首 token 时间明显恶化
- Redis / MySQL 访问异常导致大面积失败
- 会话记忆写入错误
- 权限过滤行为与 Java 不一致
- 图表 JSON、核心数字或排序出现明显回归

<!-- 下方章节继续展开当前文档的一个核心主题，阅读时建议结合实际代码一起看。 -->
## 2. 回滚原则

- 优先切回流量，不先改代码
- 不做破坏式数据库变更
- 保留 Java 服务、旧配置和旧路由
- 回滚目标应在 1 分钟内恢复可用

<!-- 下方章节继续展开当前文档的一个核心主题，阅读时建议结合实际代码一起看。 -->
## 3. 推荐回滚顺序

1. 在网关层关闭 Python 灰度 Header 或路径转发
2. 所有流量恢复到 Java 服务
3. 保留 Python 实例只读观察，不立刻清理
4. 记录异常时间点、请求样本、模型错误信息
5. 用 `scripts/smoke_test.py` 验证 Java 已恢复正常
6. 排查后再决定是否重新灰度

## 4. 数据侧注意事项

- `sa_chat_memory` 与业务表不做破坏式迁移
- 若 Java / Python 共享会话历史，应先确认 JSON 兼容
- 如灰度期间采用不同 `sessionId` 前缀，回滚时无需做数据回写

## 5. 回滚后检查项

- Java `/health` 正常
- Java `/agent/chat` 正常
- 前端或调用方已恢复旧入口
- Redis 键未出现异常膨胀
- 模型平台无异常限流

- 网关负责人执行切流
