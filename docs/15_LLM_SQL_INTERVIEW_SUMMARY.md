# Sales-Agent 项目 LLM SQL 优化总结报告

## 1. 项目背景

`sales-agent` 是一个面向销售分析场景的智能问答项目。用户通过自然语言提出问题，Agent 决定调用哪些 tool，tool 再去查询订单、销售员、大区、产品等业务数据，并生成分析结论或图表。

项目最初已经具备：

- ReAct Agent 架构
- 一组固定业务 tool
- 确定性写死 SQL / ORM 查询
- 后续引入的 LLM SQL 生成能力

本次工作的核心，不是“从 0 到 1 做 LLM SQL”，而是把一个**能跑但偏重的 LLM SQL 方案**，优化成一个更接近生产落地的版本。

---

## 2. 优化前的 LLM SQL 是什么状态

优化前，项目里很多查询都会进入 LLM SQL 链路：

1. tool 调用 `SalesQueryService`
2. `SalesQueryService` 调 `LlmSqlQueryService`
3. LLM 根据 task_type、schema、prompt 生成 SQL
4. 程序校验 SQL
5. 注入权限条件
6. 执行 SQL
7. 返回结果

这个方案的优点是灵活，但存在几个明显问题：

### 2.1 每次都要调模型，成本高

很多本来口径固定的查询，比如：

- 销售额汇总
- 趋势
- 环比/同比
- 图表底层查询
- 异常检测内部子查询

其实 SQL 结构非常稳定，但优化前仍然频繁走模型生成。

### 2.2 耗时主要花在“生成 SQL”，不是数据库

从之前的全量评估看，优化前：

- LLM 总耗时：`473563.35 ms`
- LLM 总 token：`108889`

真正慢的不是数据库执行，而是模型推理和 prompt/token 消耗。

### 2.3 异常检测成本极高

异常检测原来会拆成多个子任务，还包含循环查询：

- 按大区循环查订单数
- 按产品循环查最后出单时间
- 按销售员循环查业绩

如果这些也走 LLM SQL，就会把“生成 SQL”这件事重复很多次。

### 2.4 可观测性不够

优化前虽然能知道 LLM SQL 跑了，但不够清楚：

- 到底有没有真正调模型
- 是不是命中缓存
- 是不是命中模板
- 最终 SQL 从哪里来的

这对线上分析和面试说明都不够友好。

---

## 3. 优化目标

这次优化的目标不是“让所有查询都走 LLM SQL”，而是：

### 3.1 保留灵活性

对适合开放扩展的场景，仍然保留 LLM SQL 的能力。

### 3.2 降低耗时和 token

尽量减少不必要的模型调用。

### 3.3 提高结果稳定性

让高频、固定口径查询尽量走更稳定的路径。

### 3.4 提高可观测性

让我们能明确知道每个场景是：

- `bypass`
- `template`
- `cache`
- `generator`

---

## 4. 具体怎么优化的

本次优化不是单点优化，而是做了多层收敛。

### 4.1 operation 白名单分流

在 `SalesQueryService` 里增加了 operation 白名单控制。

核心思想：

- 不是所有查询都允许进入 LLM SQL
- 对高确定性场景，直接走写死 SQL
- 对有扩展价值的场景，才进入 LLM SQL

这样做的收益非常直接：  
先把不该进模型的查询挡在门外。

### 4.2 ranking 类改成模板化 SQL

给以下高频稳定查询加了受控模板：

- `get_top_reps`
- `get_region_ranking`
- `get_top_products`

它们现在优先命中模板，而不是每次让模型生成。

这一步非常关键，因为它把“LLM SQL”从“自由生成 SQL”收敛成了“受控模板链路”。

### 4.3 SQL 结构缓存

对于仍然需要生成 SQL 的任务，增加了 SQL 结构缓存。

也就是说：

- 第一次生成 SQL 后缓存模板结构
- 后续相同任务直接复用

这样避免了重复调模型。

### 4.4 Prompt 裁剪

不是每个 task 都喂给模型完整 schema，而是按 task_type 提供更小的 schema 和指标口径。

收益：

- 减少 token
- 减少 prompt 噪音
- 提高生成稳定性

### 4.5 异常检测批量聚合化

把异常检测从“逐个 region/product/rep 查询”改成“批量聚合查询”。

例如：

- 大区订单量 -> 一次批量查 map
- 产品最后出单时间 -> 一次批量查 map
- 销售员周期销售额 -> 一次批量查 map

这一步把异常检测从大量细碎查询收成了少量聚合查询，代码也更干净。

### 4.6 更细的 SQL 审计日志

增加了结构化 SQL 审计日志，记录：

- `tool_name`
- `task_type`
- `source`
- `template_id`
- `task_signature`
- `token`
- `generation_ms`
- `execution_ms`
- `sql`
- `params`
- `row_count`

这使得后续分析“为什么慢”“有没有调模型”“最后 SQL 是什么”变得很直接。

### 4.7 清理冗余旧链路

异常检测批量化后，把旧的逐对象单点查询链路也清掉了：

- `query_last_order_date`
- `query_order_count`
- `sum_amount_by_rep`

以及对应旧 anomaly task 的 prompt/schema 残留。

这一步不是性能优化本身，但它让代码更容易维护，也更适合面试里讲“重构闭环”。

## 5. 优化成果

### 5.1 优化前 vs 当前版本

数据来源：

- 优化前：`logs/sql_mode_compare_results_before_opt.json`
- 当前版本：`logs/sql_mode_compare_results.json`

| 指标 | 优化前 | 当前版本 | 变化 |
| --- | --- | --- | --- |
| 总场景数 | 28 | 28 | +0 |
| 结果完全一致 | 26 | 26 | +0 |
| 写死 SQL 总耗时 | 2468.79 ms | 3838.4 ms | 运行环境波动 |
| LLM 总耗时 | 473563.35 ms | 71160.57 ms | 下降约 85.0% |
| LLM 总 token | 108889 | 11748 | 下降约 89.2% |

说明：

- 静态 SQL 总耗时受本次运行环境波动影响较大，因此不要过度解读绝对值。
- 但 **LLM 侧耗时和 token 大幅下降** 这个结论非常稳定。

### 5.2 当前命中分布

| 模式 | 场景数 | 含义 |
| --- | --- | --- |
| `template` | 14 | 直接命中 SQL 模板，不调用模型 |
| `generator` | 11 | 仍然由模型真实生成 SQL |
| `generator+cache` | 2 | 首次生成，后续命中缓存 |
| `bypass` | 1 | 直接回写死 SQL |

这个分布很有价值，因为它说明：

- 现在的 LLM SQL 已经不是“全量自由生成”
- 它开始具备分层治理能力

## 6. 哪些场景优化效果最好

### 6.1 ranking 及其衍生图表

优化后这些场景几乎都很好：

- `get_top_reps`
- `get_region_ranking`
- `get_top_products`
- `generate_bar_chart`
- `generate_pie_chart`

原因：

- 模板足够稳定
- 查询口径清晰
- 结果一致性高
- token 接近 `0`

### 6.2 仍然建议保守处理的场景

以下场景目前仍不建议主推 LLM SQL：

- `query_sales_data`
- `get_sales_summary`
- `get_monthly_trend`
- `generate_line_chart`
- `detect_sales_anomalies`

原因分别是：

- 有的还在大量走 `generator`
- 有的收益不大
- 有的仍存在结果偏差
- 有的虽然能跑，但模型成本不划算

## 6.5 与“只用写死 SQL 方案”的对比

为了让这次优化更容易在面试里讲清楚，需要把当前混合方案和“完全只用写死 SQL”单独对比。

### 6.5.1 如果项目完全只用写死 SQL，会有什么优点

只用写死 SQL 的优点很明确：

- 性能最好  
  不需要任何模型推理，链路最短。

- 成本最低  
  token 消耗为 `0`。

- 稳定性最好  
  SQL 结构固定，不会因为模型输出波动而变化。

- 排障最简单  
  查问题时只需要看代码和数据库，不需要额外分析 prompt、模型输出和校验链路。

- 更适合固定 BI 类查询  
  对汇总、趋势、环比、同比、异常检测这类场景非常合适。

### 6.5.2 如果项目完全只用写死 SQL，会有什么缺点

只用写死 SQL 也有明显上限：

- 扩展新分析需求需要持续改代码  
  每新增一个维度、一个组合过滤口径、一个聚合方式，都要改 Repository / Service / Tool。

- 容易出现“查询模板越堆越多”  
  很多 ranking / chart / summary 查询本质相似，但会散落成很多手写逻辑。

- 对开放式分析支持差  
  当用户问法越来越灵活时，纯写死 SQL 的维护成本会逐步升高。

- 很难演进成统一的查询治理层  
  所有查询都各写各的，后面想加统一权限、统一模板、统一审计，会更分散。

### 6.5.3 当前混合方案和纯写死 SQL 的本质区别

当前方案不是“放弃写死 SQL”，而是把系统拆成三层：

1. 写死 SQL  
   负责固定、高频、强稳定性的查询。

2. 模板 SQL  
   负责 ranking 这类高频稳定聚合场景。

3. LLM SQL  
   负责仍需要灵活性的查询。

也就是说，当前方案吸收了纯写死 SQL 的稳定性优点，同时保留了后续扩展能力。

### 6.5.4 面试里该怎么评价这两种方案

我建议你不要把“只用写死 SQL”说成落后方案，而是这样表达：

> 如果项目目标只是固定 BI 查询，纯写死 SQL 是完全合理，甚至是最优选择。  
> 但如果希望系统具备后续扩展能力，尤其是 ranking、组合过滤和开放式分析能力，那么更好的做法不是全量 LLM，而是写死 SQL、模板 SQL、LLM SQL 的混合分层方案。

这样说会显得你的判断更成熟。

### 6.5.5 结合本项目的最终工程判断

针对 `sales-agent` 这个项目，我的判断是：

- **纯写死 SQL**  
  适合：
  - `get_sales_summary`
  - `get_monthly_trend`
  - `calc_month_over_month`
  - `calc_year_over_year`
  - `generate_line_chart`
  - `detect_sales_anomalies`
  - `query_sales_data`

- **模板 SQL / 轻量 LLM SQL**  
  适合：
  - `get_top_reps`
  - `get_region_ranking`
  - `get_top_products`
  - `generate_bar_chart`
  - `generate_pie_chart`

所以从结果上看，这次优化最后并没有证明“LLM 一定比写死 SQL 更好”，而是证明了：

> 在真实工程里，纯写死 SQL 和 LLM SQL 并不是互斥关系。  
> 更合理的落地方式，是让写死 SQL 承担稳定主链路，让模板/LLM SQL 承担灵活扩展层。

## 7. 仍然存在的两个偏差点

当前版本对比里还有两个结果偏差，这一点在面试里反而是加分项，因为说明你有工程诚实度。

### 7.1 `query_sales_data`

问题：

- 写死 SQL：查全量订单后，在文本层截前 N 条
- LLM SQL：在 SQL 层直接 `LIMIT`

结果：

- “共找到多少条订单”的口径不同

### 7.2 普通销售员 `get_monthly_trend`

问题：

- 静态逻辑当前仍按 `region_id` 查区域趋势
- LLM SQL 经过权限注入后按 `rep_id` 查个人趋势

结果：

- 返回的统计范围不同

这两个偏差说明：

- LLM SQL 不只是带来风险
- 它也能帮助暴露原有静态逻辑中的权限或口径问题

## 8. 这次优化最值得在面试里讲的点

如果你拿这去面试，我建议重点讲下面几个点。

### 8.1 你不是“无脑上 LLM”，而是在做架构分层

可以这样讲：

> 我没有把所有查询都强行切到 LLM SQL，而是把查询分成三层：  
> 固定查询走写死 SQL，稳定高频聚合走模板 SQL，真正需要灵活性的场景才走 LLM 生成 SQL。

这是一个很成熟的工程判断。

### 8.2 你优化的核心不是 prompt，而是减少模型出场次数

可以这样讲：

> 最有效的优化并不是反复调 prompt，而是从架构上减少模型调用次数，比如白名单分流、模板化和缓存。

这比单纯说“我调了 prompt”高级很多。

### 8.3 你做了性能、稳定性、可观测性三件事

可以这样拆：

- 性能：模板、缓存、批量聚合
- 稳定性：白名单、权限注入、确定性回退
- 可观测性：结构化审计日志、比较脚本、命中形态统计

### 8.4 你不仅新增能力，还回收了历史冗余代码

可以这样讲：

> 批量聚合稳定后，我还把旧的单点异常检测查询链路删掉了，避免代码同时维护两套路由。

这会让面试官感觉你不是只会“往上堆功能”，而是会把系统收干净。

## 9. 面试时可以直接说的成果话术

你可以直接用下面这段：

> 我在一个销售分析 Agent 项目里负责 LLM SQL 优化。  
> 最初很多查询都会直接走模型生成 SQL，虽然灵活，但性能和成本都比较高。  
> 我做了三类优化：  
> 第一类是分流，把固定口径查询挡回写死 SQL；  
> 第二类是模板化和缓存，把 ranking 这类高频稳定查询从自由生成改成受控模板；  
> 第三类是异常检测批量聚合化和审计日志增强。  
> 优化后，整体 LLM 耗时从 47 万毫秒降到 7.1 万毫秒，下降约 85%；  
> token 从 10.9 万降到 1.17 万，下降约 89%。  
> 同时我还能清楚区分每个场景到底是 bypass、template、cache 还是 generator。  
> 这让我最后把系统收敛成了一个更适合生产落地的混合方案，而不是全量依赖模型。

## 10. 我的工程判断总结

这次优化最后得到的核心结论不是“LLM SQL 好”或者“写死 SQL 好”，而是：

### 10.1 真实生产里应该混合使用

- 写死 SQL：负责稳定、高频、固定口径查询
- 模板 SQL：负责高频稳定但希望统一治理的聚合类查询
- LLM SQL：负责真正需要灵活性的分析查询

### 10.2 LLM SQL 最适合作为“增强层”

它更适合做：

- 灵活分析
- 新维度扩展
- 聚合类开放问法

而不适合直接承担所有主链路基础查询。

### 10.3 最终目标不是“让模型生成更多 SQL”

而是：

- 让模型只出现在它真的有价值的地方
- 让系统整体更快、更稳、更可解释

## 12. 一句话结论

这次工作的价值，不只是把 LLM SQL 跑通，而是把它从“能工作”优化成了“有边界、可治理、可观测、适合落地”的版本。
