# sale-agent 项目面试准备文档

---

## 一、项目概述

基于 **LangGraph** 的销售数据分析 AI Agent，用户用自然语言提问（趋势、排名、异常），系统通过 **ReAct（推理-行动）工作流** 调用工具查询数据库，返回文字回答 + ECharts 可视化图表。

**技术栈**：Python / FastAPI + LangGraph 1.1 + SQLAlchemy 2.0 + MySQL + Redis + Vue 3 + ECharts + Vite

**核心能力**：销售订单查询 | 排行榜 | 月度趋势 | 同环比 | 折线/柱状/饼图 | 异常检测（4 类） | 多轮对话记忆 | SSE 流式输出

---

## 二、记忆模式

### 2.1 整体策略

**"全量持久化 + 窗口注入 + 智能摘要"**——所有对话完整存 MySQL，每次只注入一个滑动窗口到 LLM 上下文，超出窗口的历史用 LLM 做智能摘要压缩。

### 2.2 存储模型（极简设计）

`sa_chat_memory` 表只有 4 个字段：

| 字段 | 类型 | 说明 |
|------|------|------|
| `id` | BIGINT PK | 主键 |
| `session_id` | VARCHAR(100) UNIQUE | 会话 ID，一个会话一行 |
| `messages` | LONGTEXT | JSON 数组，存完整对话 `[{role, content, artifacts}]` |
| `updated_at` | DATETIME | 自动更新 |

设计要点：**UPSERT 模式**，每次保存 JSON 覆盖整列，不做增量追加，一次 IO 拿到完整历史。

### 2.3 记忆加载流程

```
用户发消息 → load_memory 节点
              │
              ├─ 从 MySQL 读取完整历史 (按 session_id)
              │
              ├─ 判断：历史长度 > 20 条？(keep_recent 窗口)
              │   ├─ 否 → 全部保留，不压缩
              │   └─ 是 → 对超窗口部分做摘要压缩
              │         ├─ 主策略：LLM 压缩（需 >= 8 条才触发）
              │         └─ 降级策略：规则截断
              │
              ├─ 判断：是否追问？(关键词匹配)
              │   ├─ 独立问题 → react_messages 只放当前问题
              │   └─ 追问 → react_messages 放最近 10 条
              │
              └─ 输出：{ messages(全量), react_messages(窗口), summary_context }
```

关键逻辑：独立问题不注入任何历史到 LLM 上下文，节省 token。只有"继续"、"上面提到的"、"这个呢"等追问才注入最近 10 条。

### 2.4 LLM 智能摘要机制

**① LLM 摘要（主策略）**
- 使用独立的 LLM 实例（temperature=0.0，可配专用模型）
- 触发条件：历史消息 > 窗口(20条) **且** 超出部分 >= 8 条
- Prompt 要求输出 4-8 行中文编号摘要，只保留：已确认的分析对象、时间范围、区域、人员、TopN、图表类型、已查到的关键结论/异常、未完成的持续请求
- 如果和历史无关，输出 "无"
- 摘要结果注入到 System Prompt 的 `{history_summary}` 占位符

**② 规则摘要（降级）**
- LLM 调用失败或未触发时自动降级
- 取超窗口的最后 6 条，每条截断 80 字，拼成简单文本

### 2.5 记忆持久化

LangGraph 的流式图**不包含 persist_memory 节点**（为了 SSE 流式体验），而是在流结束后手动保存：

```
流式输出完成 → _save_reply()
                ├─ 取 state["messages"] (全量历史)
                ├─ 追加 { role: "user", content: 当前问题 }
                ├─ 追加 { role: "assistant", content: 最终回答, artifacts: [图表] }
                └─ MySQL UPSERT
```

异常时也会保存错误回复到记忆，防止丢对话。

### 2.6 记忆配置参数

| 参数 | 默认值 | 作用 |
|------|--------|------|
| `AGENT_MEMORY_WINDOW_MESSAGES` | 20 | 上下文窗口大小 |
| `AGENT_MEMORY_FOLLOWUP_LIMIT` | 10 | 追问时最多注入条数 |
| `AGENT_MEMORY_LLM_SUMMARY_ENABLED` | True | LLM 摘要开关 |
| `AGENT_MEMORY_LLM_SUMMARY_TRIGGER_MESSAGES` | 8 | 触发 LLM 摘要的最小超窗口条数 |
| `AGENT_MEMORY_SUMMARY_MESSAGES` | 6 | 规则摘要采样的条数 |
| `AGENT_MEMORY_SUMMARY_CHARS` | 80 | 摘要每条最大字符数 |
| `AGENT_MEMORY_LLM_SUMMARY_MODEL` | 空(复用主模型) | 摘要专用模型 |

### 2.7 记忆模式面试话术

> "我们采用分层记忆策略：完整历史存 MySQL，LLM 上下文只注入必要窗口，既保证信息不丢，又控制 token 成本。超出窗口的历史用独立 LLM 做智能摘要压缩，保留语义意图而非原文。追问识别基于关键词匹配，独立问题不注入历史，避免干扰。LLM 摘要失败自动 fallback 到规则摘要，保证系统弹性。"

## 三、Agent 编排 (LangGraph)

### 3.1 图拓扑

```
START
  │
  ▼
[bootstrap]           ← 初始化状态
  │
  ▼
[load_memory]         ← 加载/压缩对话历史
  │
  ▼
[guardrails]          ← 关键词硬拦截
  │
  ├──(rejected)──→ [persist_memory] → END
  │
  └──(pass)──→ [react_agent] ←→ [react_tool_executor]
                   │                      ▲
                   └──(done)──→ [persist_memory] → END
```

共 7 个节点，`react_agent` ↔ `react_tool_executor` 形成 ReAct 循环。流式图省略 `persist_memory` 节点，持久化在流结束后手动完成。

### 3.2 核心设计：状态与上下文分离

| 概念 | 类型 | 作用 | 生命周期 |
|------|------|------|---------|
| `GraphState` | TypedDict | 对话数据：消息、答案、图表、工具结果 | 单次请求 |
| `AgentContext` | TypedDict | 基础设施：工具实例、LLM、中间件 | 单次请求，通过 LangGraph Runtime 的 `context` 传递 |

数据流和基础设施解耦，节点只操作 State，Context 提供能力。

### 3.3 ReAct 循环机制

```
react_agent 节点：
  1. 构建 SystemMessage（日期 + 用户信息 + 历史摘要 + 当前问题）
  2. 确保 llm_with_tools 已缓存（temperature=0.2，绑定 12 个工具）
  3. LLM.invoke() → AIMessage（可能包含 tool_calls）
  4. _finalize_react_turn() 决策：
     ├─ 有 tool_calls + 未超限 → 返回 final_answer=None（继续循环）
     ├─ 无 tool_calls → 提取内容作为 final_answer（退出循环）
     └─ 工具调用次数超限 → 强制退出，生成兜底回答

react_tool_executor 节点：
  1. 遍历 AIMessage 中的 tool_calls
  2. 逐个执行工具（try/except 包裹，单工具失败不崩溃）
  3. 结果包装为 ToolMessage 追加到 react_messages
  4. 无条件回到 react_agent（静态边）
```

### 3.4 终止条件（三层防护）

1. **自然终止**：LLM 返回无 tool_calls 的消息
2. **硬上限**：`agent_max_tool_calls`（默认 8），在 `_finalize_react_turn` 和 `react_tool_executor_node` 两处检查
3. **空响应保护**：LLM 无输出时，用 `tool_results.text` 或默认文本兜底

### 3.5 流式图 vs 非流式图

| | 非流式图 | 流式图 |
|------|----------|--------|
| `persist_memory` 节点 | 有 | 无 |
| 执行方式 | `graph.invoke()` 一次调用 | 手动编排：逐个调用节点 + interleave SSE yield |
| 记忆保存 | persist_memory 节点内完成 | `SalesAgent._save_reply()` 手动保存 |
| 中间反馈 | 无 | 状态事件 + token 流 |

为什么流式图不用 LangGraph 原生的 `.stream()`？因为需要更细粒度的控制——在节点之间插入状态事件，以及 token 级别的流式输出。

### 3.6 懒加载缓存

以下对象在单次请求中懒加载并缓存在 `runtime.context` 中：

- `llm_with_tools`（绑定了工具的 LLM 实例）
- `react_tools`（12 个结构化工具列表）
- `tool_map`（工具名 → 工具的查找字典）
- `middleware`（AgentMiddleware 实例）
- `memory_summary_llm`（摘要专用 LLM 实例）

避免在 ReAct 循环的每一轮重复创建。

### 3.7 Agent 编排面试话术

> "我们用 LangGraph 的 StateGraph 实现了标准的 ReAct 模式。Agent 在 react_agent 节点做推理决策，在 react_tool_executor 节点执行工具，两个节点形成循环。终止条件有三层保障：LLM 自然结束、最大工具调用次数硬限制、空响应兜底。状态和上下文分离，数据流走 State，基础设施走 Context。流式模式下，我们手动编排节点执行，在节点间插入进度事件和 token 流，给用户实时反馈。关键组件都做了请求级懒加载缓存，避免 ReAct 循环中重复创建。"

## 四、工具调用系统

### 4.1 12 个工具函数

| 类别 | 工具名 (LLM 可见) | 核心参数 | 返回 |
|------|------------------|---------|------|
| 查询 | `query_sales_data` | start_date, end_date, region, rep, limit | 订单明细文本 |
| 汇总 | `get_sales_summary` | start_date, end_date, region | 销售总额 |
| 排名 | `get_top_reps` | start_date, end_date, region, top_n | 销售员排名 |
| 排名 | `get_region_ranking` | start_date, end_date | 区域排名 |
| 排名 | `get_top_products` | start_date, end_date, top_n, region | 产品排名 |
| 对比 | `calc_month_over_month` | current_start, current_end, 基期(可选), region | 环比分析 |
| 对比 | `calc_year_over_year` | start_date, end_date, region | 同比分析 |
| 趋势 | `get_monthly_trend` | months, region | 月度趋势 |
| 图表 | `generate_line_chart` | months, region, title | 折线图 ECharts JSON |
| 图表 | `generate_bar_chart` | dimension, start/end, title, top_n | 柱状图 ECharts JSON |
| 图表 | `generate_pie_chart` | dimension, start/end, title, top_n | 饼图 ECharts JSON |
| 异常 | `detect_sales_anomalies` | 无 | 异常报告文本 |

### 4.2 工具封装方式

使用 `StructuredTool.from_function()` 包装：

```python
StructuredTool.from_function(
    func=tools.query_orders,          # 实际函数
    name="query_sales_data",           # LLM 看到的函数名
    description="查询销售订单明细...",  # 中文描述，LLM 选工具的依据
    args_schema=QuerySalesDataArgs,    # Pydantic 模型 → JSON Schema
)
```

9 个 Pydantic Args Schema，每个字段带 `Field(description=...)` 让 LLM 理解参数含义。

### 4.3 异常检测（4 类自动扫描）

| 检测类型 | 逻辑 | 严重程度 |
|----------|------|---------|
| 区域订单骤降 | 近 2 周 vs 前 4 周均值 | HIGH(>60%) / MEDIUM |
| 零销量产品 | 最近订单日期 > 阈值天数 | HIGH(>=14天) / MEDIUM(>=7天) |
| 高退款率 | 近 30 天退款率 > 15% | HIGH(>30%) / MEDIUM |
| 销售员业绩下滑 | 近 30 天 vs 前 30 天 | HIGH(>80%) / MEDIUM(>50%) |

### 4.4 输入验证

- **日期**：ISO 格式 (`date.fromisoformat`)
- **区域名称**：白名单校验（华东区、华南区、华北区、西南区）
- **维度**：白名单校验（region / rep / category / product）
- **top_n**：钳位到 [1, 20]
- **months**：钳位到 [1, 24]

### 4.5 工具调用面试话术

> "我们封装了 12 个确定性工具函数，覆盖查询、汇总、排名、趋势、图表、异常检测六大类。每个工具通过 Pydantic Schema 自动生成 JSON Schema 给 LLM，有严格的白名单参数验证。工具执行失败不会导致整个图崩溃，而是返回 ToolMessage 错误信息，让 LLM 在下一轮循环中自行纠正或尝试其他方案。"

## 五、SQL Agent（LLM 生成 SQL）

### 5.1 设计动机

预定义的 Repository SQL 覆盖了常见查询，但遇到灵活的多条件组合查询时不够用。SQL Agent 让 LLM 根据自然语言意图**生成参数化 SQL**，同时通过多层安全机制保证安全。

### 5.2 核心流程

```
用户问题 → SqlTaskSpec（结构化任务描述）
              │
              ├─ 1. 模板匹配（优先）     → 7 个预审 SQL 模板
              ├─ 2. 缓存查找             → SHA-256 签名匹配
              └─ 3. LLM 生成（重试循环）  → 校验 → 权限注入 → 执行
```

### 5.3 四级安全防护

| 层级 | 检查内容 | 实现 |
|------|---------|------|
| **关键字检查** | 禁止 INSERT/UPDATE/DELETE/DROP/TRUNCATE/CREATE/CALL/SHOW 等 18 个危险关键字；禁止分号（多语句）；禁止注释 | `SqlValidator` |
| **Schema 白名单** | 只允许 4 张表（sa_sales_order/rep/region/product）；每表指定列的列白名单；禁止 `SELECT *` | `SqlSchemaRegistry` |
| **业务规则校验** | 汇总查询必须包含 `status='COMPLETED'` 过滤；明细查询不能有该过滤；必须使用 `:start_date`/`:end_date` 参数占位符 | `SqlValidator` 第三阶段 |
| **行级权限注入** | 销售员 → 注入 `rep_id = :scope_rep_id`；经理 → 注入 `region_id = :scope_region_id` | `PermissionPolicyInjector` |

### 5.4 参数安全原则

> SQL 结构可以由 LLM 生成，但**参数值必须来自应用侧**。从 LLM 返回的 SQL 中提取 `:param` 占位符，用 `SqlTaskSpec` 中的值填参，杜绝参数注入。

### 5.5 影子模式（Shadow Mode）

`LLM_SQL_SHADOW_MODE=True` 时：
- 先用确定性 SQL 返回结果给用户
- 同时在后台静默执行 LLM SQL
- 对比结果但不影响用户体验
- 用于 **A/B 测试和灰度验证**

### 5.6 7 个预审 SQL 模板

| 模板 ID | 任务类型 | 说明 |
|---------|---------|------|
| `sales_summary_v2` | sales_summary | SUM(amount) + COMPLETED + 可选 region |
| `monthly_trend_v2` | monthly_trend | DATE_FORMAT 分组 + SUM + COUNT |
| `refund_rates_v2` | refund_rates | CASE WHEN REFUNDED + COUNT |
| `rep_ranking_v2` | rep_ranking | JOIN rep + region, SUM, DESC LIMIT |
| `region_ranking_v1` | region_ranking | JOIN region, GROUP BY |
| `product_ranking_v1` | product_ranking | JOIN product, SUM amount + quantity |

### 5.7 SQL Agent 面试话术

> "SQL Agent 是我们最有技术深度的模块之一。核心挑战是既要发挥 LLM 处理灵活查询的能力，又要保证数据库安全。我们设计了四层防护：关键字黑名单、Schema 白名单、业务规则校验、行级权限注入。最关键的安全原则是——SQL 结构可以由 LLM 生成，但参数值必须来自应用侧，杜绝参数注入风险。此外，影子模式让我们可以安全地灰度验证 LLM SQL 的准确性，在后台静默执行对比而不影响用户。"

## 六、流式输出 (SSE)

### 6.1 实现方式

不是用 LangGraph 原生的 `astream_events()`，而是**手动编排节点执行**，在节点间插入 SSE 事件：

```
1. bootstrap_node(state, runtime)
2. yield "正在读取会话上下文..."
3. load_memory_node(state, runtime)
4. yield "正在检查问题边界..."
5. guardrails_node(state, runtime)
6. ReAct 循环:
   ├─ yield token... token... (LLM stream)
   ├─ yield "正在调用工具：get_top_reps..."
   ├─ react_tool_executor_node(state, runtime)
   └─ loop
7. yield artifacts (ECharts JSON)
8. yield "[DONE]"
```

### 6.2 SSE 事件类型

| 事件 | 含义 | data 格式 |
|------|------|----------|
| `status` | 进度提示 | 中文字符串 |
| `token` | LLM 输出 token | 文本片段 |
| `artifacts` | 图表数据 | ECharts option JSON |
| `done` | 完成信号 | `[DONE]` |
| `error` | 错误信息 | 错误文本 |

### 6.3 Token 流式输出细节

- 调用 `llm_with_tools.stream()` 逐块获取
- 检测到 `tool_calls` 块时不输出（工具调用信息对用户不可见）
- 首个 text token 前自动 emit `"正在输出回答..."` 状态
- 如果 `stream()` 不可用（旧模型），自动降级到同步调用 + `stream_text_chunks()` 模拟分块

### 6.4 流式输出面试话术

> "我们没有直接用 LangGraph 的 stream，而是手动编排节点。这样可以在节点间插入'正在读取会话上下文'、'正在调用工具'等状态事件，给用户实时进度反馈。Token 流式输出用的是 LLM 原生的 stream API，工具调用信息被过滤掉，用户只看到文字回答。如果模型不支持 stream，自动降级到同步调用 + 模拟分块。"

## 七、护栏机制 (Guardrails)

### 7.1 两层防护

**第一层：关键词硬拦截（guardrails 节点）**
- 在 LLM 调用**之前**执行，零 token 成本
- 写操作关键词：`修改、删除、新增、插入、更新、写入`
- 预测关键词：`预测、预估、明年销售、未来销售`
- 匹配后直接返回拒绝消息，**跳过整个 ReAct 循环**

**第二层：System Prompt 软约束**
- REACT_SYSTEM_PROMPT 中明确声明能力边界
- "只能查询和分析销售数据，不能修改、删除或新增业务数据"
- "可以分析历史销售数据，但不能预测未来销售结果"

### 7.2 护栏面试话术

> "护栏有两层——第一层是关键词硬拦截，在 LLM 之前就挡掉写操作和预测类请求，零 token 成本，确定性 100%；第二层是 System Prompt 软约束，确保 LLM 在边界内运行。简单但有效。"

## 八、错误处理策略

### 8.1 分层容错

| 层级 | 策略 | 影响范围 |
|------|------|---------|
| **工具执行** | 单个工具失败 → ToolMessage(err) → LLM 可重试 | 仅该工具调用 |
| **LLM 摘要** | 摘要 LLM 失败 → 自动降级到规则摘要 | 仅摘要质量 |
| **Redis 缓存** | Redis 不可用 → 静默跳过缓存，直接查 DB | 性能下降 |
| **SQL Agent** | LLM SQL 生成失败 → fallback 到确定性 SQL | 仅该查询 |
| **图执行** | 整个图异常 → `build_error_state()` → 保存错误回复到记忆 | 该次请求 |
| **HTTP** | 异常处理器 → 友好的 JSON 错误响应 | 该次请求 |

### 8.2 关键设计：工具失败不崩溃

```python
try:
    result = tool.invoke(args)
except Exception as exc:
    content = f"工具执行失败：{exc}"
    # 作为 ToolMessage 返回给 LLM，让它在下轮自行纠正
    messages.append(ToolMessage(content=content, ...))
```

### 8.3 错误处理面试话术

> "我们采用分层容错，核心原则是：单个组件的失败不能导致整个请求崩溃。工具执行失败，LLM 可以在下一轮尝试其他方案；LLM 摘要失败，自动降级到规则模式；Redis 挂了，静默跳过缓存。只有整个图级别的异常才会返回错误给用户，而且一定会保存错误上下文到对话记忆中，方便排查。"

## 十一、数据库设计

### 11.1 5 张表

```
sa_sales_region    (区域)   ─┐
sa_sales_rep       (销售员) ─┼─ 维表
sa_product         (产品)   ─┘
sa_sales_order     (订单)   ←── 事实表（关联上面三张维表）
sa_chat_memory     (对话记忆) ←── session_id UNIQUE, JSON 存全量消息
```

### 11.2 技术选型

- **无 ORM 关系映射**：不用 `relationship()`，所有 JOIN 手写 SQL，避免 N+1
- **仓储模式**：每个表一个 Repository 类，封装所有查询
- **Decimal 类型**：金额字段用 `Numeric(12,2)`，Python 侧用 `Decimal`
- **查询事件钩子**：SQLAlchemy `before_cursor_execute` / `after_cursor_execute` 实现自动计时

### 11.3 启动流程

```
main.py lifespan:
  1. CREATE DATABASE IF NOT EXISTS (utf8mb4)
  2. 执行 schema.sql（5 张表 DDL，事务执行）
  3. 验证 5 张表都已创建
  4. 如果表为空或 reseed=True → 执行 data.sql 灌入演示数据
```

### 11.4 缓存策略

| 缓存类型 | 后端 | TTL | 缓存内容 |
|----------|------|-----|---------|
| 销售数据结果 | Redis | 300s | 总额、排名、趋势的查询结果 |
| 区域元数据 | Redis | 1800s | 区域名称 → ID 的映射 |
| SQL 模板 | Redis / 内存 | 3600s | LLM 生成的 SQL 结构（非结果） |

Redis 不可用时静默跳过所有缓存，直接查 DB。

### 11.5 数据库面试话术

> "5 张表，经典的星型模型——区域、销售员、产品三张维表，订单事实表关联它们，加一张对话记忆表。我们刻意不用 ORM 的 relationship，所有 JOIN 手写 SQL，这样查询性能可控，没有 N+1 问题。缓存用 Redis，但做了降级——Redis 不可用时自动跳过缓存，不阻塞业务。数据库初始化全自动化，启动时检查、建表、灌数据一条龙。"

## 十二、综合面试话术模板

> "这是一个基于 LangGraph 的销售数据分析 AI Agent，我负责全栈开发。核心架构是 ReAct 模式——LLM 推理需要调用哪个工具，执行工具获取数据，再推理下一步，直到能回答用户问题。
>
> **记忆系统**：全量对话存 MySQL，注入 LLM 时只取滑动窗口，超窗口的历史用独立 LLM 做智能摘要。追问识别基于关键词，独立问题不注入历史，节省 token。
>
> **SQL Agent**：预定义 SQL 覆盖不了的灵活查询，让 LLM 生成参数化 SQL，通过四层安全检查。关键原则是 SQL 结构可以 LLM 生成，但参数值必须来自应用侧。
>
> **流式输出**：手动编排节点执行，在节点间插入进度事件，token 级别流式输出。
>
> **分层容错**：单组件故障不传导——工具失败 LLM 可重试，缓存挂掉自动降级，LLM 摘要失败 fallback 到规则模式。
>
> **技术栈**：FastAPI + LangGraph + SQLAlchemy 2.0 + MySQL + Redis + Vue 3 + ECharts，前后端分离。带完整的测试套件（pytest，11 个测试文件）和运维脚本。"
